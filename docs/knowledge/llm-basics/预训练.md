# 预训练

预训练（Pretraining）是大语言模型获得通用语言能力的主要阶段。它通常使用大规模文本、代码和多种结构化语料，让模型学习“在给定上下文时，下一个 token 最可能是什么”。

预训练不是让模型直接学会聊天，也不是针对某个具体业务任务做优化。它更像是在给模型打底：

- 学会语言规律和文档结构。
- 学会常识、事实、代码模式和表达习惯。
- 学会在上下文中建立关联。
- 学会续写、归纳、类比和一定程度的推理模式。

预训练完成后的模型通常叫 base model。Base model 有很强的续写和生成能力，但不一定会稳定遵循人类指令。要把它变成好用的 chat model，还需要指令微调、偏好对齐、强化学习、安全评估和部署优化。

## 1. 核心目标

现代 Decoder-only LLM 最常见的预训练目标是 next token prediction：

```text
给定前面的 token，预测下一个 token。
```

例如：

```text
输入上下文: 人工智能正在改变
目标 token: 软件
```

训练时，模型会对词表里的每个 token 给出一个概率分布。正确 token 的概率越高，loss 越低。

这个目标看起来简单，但规模足够大时，它会迫使模型学习很多隐藏能力：

- 语法：什么词能接在什么词后面。
- 语义：上下文里实体、关系和指代是什么。
- 知识：常见事实和概念之间的关联。
- 格式：Markdown、JSON、代码、表格、论文、问答等结构。
- 推理模式：因果、比较、步骤分解、条件判断等文本模式。

可以把预训练理解为：模型通过预测下一个 token，被迫压缩和重建训练语料中的规律。

## 2. 自监督学习

预训练通常属于自监督学习。它不需要人工为每条样本写标签，因为标签可以从文本本身自动构造。

一段文本：

```text
大型语言模型通过预训练学习通用能力。
```

可以被拆成很多训练样本：

```text
给定: 大型
预测: 语言

给定: 大型 语言
预测: 模型

给定: 大型 语言 模型
预测: 通过
```

这就是预训练能扩展到超大规模的关键原因：只要有足够多的高质量文本，就能自动形成训练信号。

但“自动有标签”不等于“数据没有成本”。真正难的是数据收集、清洗、去重、分布控制、版权合规、安全过滤和评估隔离。

## 3. 数据来源

预训练数据通常来自多种语料混合。常见类型包括：

| 数据类型 | 价值 | 风险 |
| --- | --- | --- |
| 网页文本 | 覆盖面广，语言和知识丰富 | 噪声大、重复多、质量不稳定 |
| 书籍和长文档 | 结构完整，长程依赖更强 | 授权和版权要求高 |
| 论文和技术文档 | 专业知识密度高 | 领域偏斜，格式复杂 |
| 代码仓库 | 提升代码生成、工具使用和结构化推理 | 许可证、泄漏密钥、重复代码 |
| 问答和论坛 | 接近真实问题表达 | 错误答案、攻击性内容、低质量讨论 |
| 数学和合成数据 | 可增强推理、符号和格式能力 | 分布可能不自然，容易模板化 |
| 多语言语料 | 提升跨语言能力 | 低资源语言质量和比例难控制 |

数据配比会直接影响模型能力。代码占比更高，通常更有利于代码和结构化输出；中文语料占比更高，中文理解和生成会更稳；高质量长文档更多，模型可能更擅长长上下文表达。

## 4. 数据清洗

预训练数据规模很大，原始数据不能直接拿来训练。常见清洗步骤包括：

1. 格式解析：从 HTML、PDF、代码仓库或文档中抽取正文。
2. 质量过滤：删除乱码、模板页、广告页、机器翻译垃圾文本和低信息密度内容。
3. 语言识别：按语言分类，控制不同语言比例。
4. 安全过滤：删除明显违法、攻击性、恶意代码或敏感内容。
5. 隐私过滤：识别并删除密钥、手机号、邮箱、身份证号、内部地址等敏感信息。
6. 去重：删除完全重复和高度相似文本。
7. 文档切分：按长度、段落和语义边界切成适合训练的样本。

清洗质量会显著影响训练结果。低质量数据不是“多一点无所谓”，它会消耗训练预算，还可能让模型学到错误事实、糟糕格式和不安全行为。

## 5. 去重为什么重要

去重有两个目的：

- 提高训练效率：重复样本会浪费 token budget。
- 降低评估污染：如果评估集内容出现在训练集中，评估结果会虚高。

常见去重层级包括：

| 去重层级 | 说明 |
| --- | --- |
| 精确去重 | 删除完全相同的文档、段落或代码文件 |
| 近似去重 | 删除高度相似但有少量改写的内容 |
| 模板去重 | 删除网页导航、版权声明、页脚等重复模板 |
| benchmark 去重 | 从训练语料中移除评测集题目、答案和相似变体 |

对大模型来说，数据污染不只是“背过答案”。如果训练集中存在大量 benchmark 题目、题解或近似样本，模型可能在公开榜单上表现很好，但真实泛化能力并没有对应提升。

## 6. Tokenizer 与训练语料

模型训练前，需要先确定 tokenizer。Tokenizer 决定文本如何切成 token，也决定词表覆盖哪些语言、符号和代码片段。

同一句话在不同 tokenizer 下可能切成不同 token：

```text
文本: 解释一下 KV Cache
tokenizer A: [解释, 一下, KV, Cache]
tokenizer B: [解, 释, 一, 下, K, V, Cache]
```

Tokenizer 会影响：

- 中文、英文、代码和符号的压缩效率。
- 同样上下文窗口能容纳多少信息。
- 训练和推理成本。
- 低资源语言和专业术语的处理效果。
- 代码缩进、空格、特殊符号的稳定性。

预训练通常会先在代表性语料上训练 tokenizer，再用这个 tokenizer 把全部训练数据转换成 token ids。Tokenizer 一旦确定，后续训练和部署都要保持一致。

## 7. 训练过程

一个简化的预训练流程如下：

```text
原始语料
  -> 清洗、过滤、去重
  -> tokenizer 编码
  -> 组成固定长度 token 序列
  -> 模型前向计算 logits
  -> 计算 next token loss
  -> 反向传播更新参数
  -> 周期性评估和保存 checkpoint
```

训练时通常不会只看单条文本，而是把大量 token 打包成批次，在 GPU 集群上并行训练。模型参数、优化器状态、梯度、激活值和数据加载都会消耗大量显存、内存、存储和网络带宽。

训练大模型的工程重点包括：

- 数据吞吐是否跟得上 GPU。
- 多机多卡通信是否稳定。
- checkpoint 是否能可靠保存和恢复。
- loss 是否平稳下降。
- 是否出现梯度爆炸、NaN 或训练发散。
- 不同数据源的采样比例是否按计划执行。

预训练不是“把数据扔进去等结果”。它更像一个长周期工程系统，需要持续监控数据、训练稳定性、评估指标和基础设施状态。

## 8. Loss 与 Perplexity

预训练最常见的损失函数是交叉熵损失。直观理解：

```text
正确 token 的预测概率越高，loss 越低。
```

如果模型认为正确 token 概率很低，就会受到更大惩罚。

Perplexity 可以看作 loss 的另一种表达。它衡量模型面对一段文本时有多“困惑”。通常：

- loss 越低，perplexity 越低。
- perplexity 越低，模型越容易预测这段文本。

但低 loss 不等于模型一定好用。原因包括：

- 训练集 loss 低可能只是记住了训练数据。
- 通用 loss 下降不一定带来特定业务能力提升。
- 聊天、工具调用、安全拒答和事实引用需要专项评估。
- 数据污染会让 benchmark loss 或准确率虚高。

## 9. Scaling Law 的直觉

Scaling law 讨论的是模型规模、数据规模、计算量和性能之间的关系。

直觉上，模型能力通常受三件事共同影响：

- 参数量：模型能存储和表达多少模式。
- 训练 token 数：模型见过多少高质量样本。
- 计算量：训练过程能进行多少有效更新。

如果模型很大但数据太少，容易欠训练或记忆数据；如果数据很多但模型太小，模型容量可能不够；如果计算量不足，训练还没收敛就停了。

所以预训练不是单纯追求“参数越大越好”。更合理的问题是：

```text
在给定预算下，模型大小、数据 token 数和训练步数怎样配比最划算？
```

对工程团队来说，scaling law 的实际意义是帮助做预算决策：

- 继续扩大模型，还是先扩大高质量数据？
- 训练一个更大的模型，还是把中等模型训练得更充分？
- 继续预训练能不能带来足够收益？
- 计算预算应该投到预训练、后训练还是评估数据上？

## 10. 预训练学到了什么

预训练学到的不是一个显式知识库，而是参数中的统计规律和表示能力。

可以分几层理解：

| 层次 | 模型学到的内容 |
| --- | --- |
| 字符和 token 模式 | 拼写、标点、分词、代码符号 |
| 局部语言规律 | 词语搭配、语法结构、短句表达 |
| 长程依赖 | 指代、主题延续、段落结构 |
| 世界知识 | 常见事实、概念关系、实体属性 |
| 格式模式 | Markdown、JSON、表格、论文、代码 |
| 任务模式 | 翻译、总结、问答、解释、分类的文本模式 |
| 推理模式 | 分步骤分析、类比、条件判断、数学和代码模式 |

需要注意的是，模型“知道”某些内容，并不等于它能稳定、可验证地调用这些内容。预训练模型生成的是概率上合理的文本，不是从数据库中检索事实。

这也是幻觉问题的根源之一：当模型缺乏可靠依据时，它仍然可能生成看起来合理的内容。

## 11. Base Model 与 Chat Model

预训练后得到的 base model 主要擅长续写。给它一段文本，它会延续这个文本的分布。

例如输入：

```text
用户：请解释什么是 KV Cache。
助手：
```

Base model 可能继续写答案，也可能继续补全对话、写成文章、重复格式，甚至生成不符合预期的内容。它未必真正理解“应该作为助手回答用户”。

Chat model 通常还经过后训练：

```text
预训练 base model
  -> 指令微调
  -> 偏好优化 / 强化学习
  -> 安全与格式评估
  -> chat model
```

后训练让模型学会：

- 区分 system、user、assistant 等角色。
- 按指令完成任务。
- 控制语气、格式和长度。
- 拒绝不安全请求。
- 在多轮对话中保持一致。

所以，不要把“预训练能力强”和“产品可用”混为一谈。预训练提供基础能力，后训练决定交互方式和行为边界。

## 12. 预训练、微调和强化学习的区别

| 阶段 | 主要目标 | 数据形式 | 训练信号 | 典型产物 |
| --- | --- | --- | --- | --- |
| 预训练 | 学通用语言、知识和模式 | 大规模原始文本、代码、文档 | next token loss | Base model |
| 指令微调 / SFT | 学会按指令完成任务 | 指令 + 标准回答 | 模仿高质量答案 | Instruct model |
| 偏好优化 / 强化学习 | 让回答更符合偏好或奖励 | chosen/rejected、reward、verifier | 偏好或奖励信号 | 更对齐的 chat model |
| 继续预训练 | 补充领域语料分布 | 大量领域文本 | next token loss | 领域 base 或中间模型 |

如果目标是让模型熟悉大量领域术语和文档风格，可以考虑继续预训练。如果目标是让模型按固定格式回答，通常优先考虑 SFT。如果目标是让模型在多个可用答案中更偏向“更好”的那个，再考虑偏好优化或强化学习。

## 13. 继续预训练

继续预训练（Continued Pretraining）是在已有 base model 上，用新的大规模语料继续做语言建模训练。

它常用于：

- 让模型熟悉某个行业的术语和表达。
- 增强代码、数学、法律、医学、金融等领域分布。
- 增强某种语言或低资源语言能力。
- 在 SFT 前先补充领域基础。

继续预训练和微调容易混淆：

| 方式 | 适合解决的问题 |
| --- | --- |
| 继续预训练 | 模型不熟悉领域语料、术语、文档风格 |
| SFT | 模型不知道应该怎样完成任务或输出格式 |
| RAG | 模型需要访问最新、私有、可追溯的知识 |

如果业务资料经常变化，继续预训练通常不是首选。更合适的方式可能是 RAG，因为外部知识库更容易更新、审计和回滚。

## 14. 数据污染与评估泄漏

数据污染是预训练中最容易被低估的问题之一。

常见污染形式包括：

- benchmark 题目或答案进入训练集。
- 评估集的改写版本进入训练集。
- 测试集中使用的业务样本被用于继续训练。
- 在线用户反馈没有隔离，回流到了下一轮训练集。
- RAG 检索语料和评估答案之间存在重叠。

污染会导致模型看起来更强，但上线后效果不稳定。尤其是公开 benchmark，很容易因为网页、题解、论坛讨论和代码仓库而泄漏。

降低污染风险的做法：

1. 在训练前固定评估集。
2. 对训练集和评估集做精确去重与近似去重。
3. 按时间、来源、客户或项目切分数据。
4. 对公开 benchmark 做专项过滤。
5. 保留数据版本、过滤规则和样本来源记录。

评估可信度和模型能力同样重要。没有干净评估集，就很难判断一次训练到底带来了真实提升，还是只是记住了测试答案。

## 15. 常见风险

### 15.1 数据质量不足

数据越大，问题也越容易被放大。低质量网页、重复模板、错误答案和混乱格式会降低训练效率。

处理方式：

- 提高高质量数据占比。
- 对不同来源设置质量过滤规则。
- 做抽样人工检查。
- 监控不同数据源上的 loss 和评估表现。

### 15.2 数据偏差

训练语料来自哪里，模型就更像哪里。如果某类语言、地域、行业或观点被过度代表，模型输出也会带有偏差。

处理方式：

- 控制语种、领域和来源配比。
- 增加多样化数据。
- 在评估集中覆盖不同人群、场景和表达方式。

### 15.3 版权与授权

预训练数据可能涉及版权、许可证和使用限制。代码数据还要关注开源许可证兼容性。

处理方式：

- 记录数据来源和授权信息。
- 对敏感来源建立排除规则。
- 区分可训练、可检索、可展示的数据权限。

### 15.4 隐私和密钥泄漏

网页、日志、代码仓库和文档中可能包含个人信息、内部 token、API key 或未公开资料。

处理方式：

- 训练前做 PII 和密钥扫描。
- 对代码仓库做 secret filtering。
- 对输出做敏感信息回归测试。
- 对内部数据建立脱敏流程。

## 16. 实践建议

如果要做预训练或继续预训练，可以按以下顺序推进：

1. 先明确目标：通用模型、领域模型、代码模型、多语言模型还是继续预训练。
2. 先建立评估集：包括通用能力、目标领域能力、安全和格式能力。
3. 先做数据审计：确认来源、授权、质量、重复率和敏感信息风险。
4. 先小规模试训：验证 tokenizer、数据管道、loss 曲线和 checkpoint 恢复。
5. 再扩大规模：逐步扩大 token 数、模型规模和训练时长。
6. 持续比较基线：不要只看训练 loss，要看固定评估集和人工抽检。
7. 保留可复现记录：数据版本、配比、代码版本、随机种子、训练参数和 checkpoint。

对大多数应用团队来说，从零预训练通常成本过高。更常见的路径是：

```text
选择合适开源或商用基座模型
  -> 用 RAG 接入可更新知识
  -> 用 SFT 固化任务格式
  -> 用偏好优化改善回答质量
  -> 用评估体系持续回归
```

## 17. 总结

预训练是大语言模型能力的基础。它通过大规模自监督学习，让模型在预测下一个 token 的过程中学到语言、知识、格式、代码和推理模式。

理解预训练时，可以抓住几条主线：

- 核心目标是 next token prediction。
- 数据质量、配比和去重决定训练上限。
- Tokenizer 会影响语料压缩效率、上下文容量和多语言表现。
- Loss 和 perplexity 是训练指标，但不能替代真实任务评估。
- Base model 不等于 chat model，后训练负责指令跟随和偏好对齐。
- 数据污染会让评估虚高，必须从数据阶段控制。
