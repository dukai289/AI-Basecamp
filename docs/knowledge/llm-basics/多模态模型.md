# 多模态模型

多模态模型是能同时处理文本、图片、视频、音频等不同类型输入的模型。对大语言模型来说，最常见的形态是 VLM（Vision Language Model）：把图片或视频转成模型能理解的视觉表示，再交给语言模型做问答、推理和生成。

可以先用一句话理解：

```text
多模态模型 = 把非文本内容转成 token 或向量，再接入 LLM 的上下文。
```

典型图文模型结构如下：

```text
图片
  -> 图像预处理
  -> vision encoder
  -> projector / adapter
  -> visual tokens
  -> LLM
  -> 文本回答
```

它不是让 LLM 直接“看见”图片，而是通过视觉模块把图片压缩成一组表示，让语言模型在生成时可以注意到这些表示。

## 1. 多模态模型解决什么问题

纯文本 LLM 只能处理已经变成文字的信息。现实任务里，大量信息并不天然是纯文本：

- 截图、照片、票据、表格图片。
- 图表、流程图、架构图、白板照片。
- 视频片段、监控画面、操作录屏。
- 语音、会议录音、环境声音。
- 图文混排的网页、PDF、PPT。

多模态模型的目标是让模型能在这些输入上完成任务，例如：

| 任务 | 例子 |
| --- | --- |
| OCR 与文档理解 | 读取发票、合同截图、表格图片 |
| 视觉问答 | 根据图片回答“这是什么”“哪里异常” |
| 图表理解 | 解读柱状图、折线图、仪表盘截图 |
| 多图对比 | 对比两张产品图、找 UI 差异 |
| 视频理解 | 总结视频内容、识别关键事件 |
| 语音交互 | 听懂语音并用语音回答 |
| 具身智能 | 根据视觉观察规划机器人动作 |

很多应用不是单纯“识图”，而是图像、文本、业务规则和推理混合在一起。例如，用户上传一张报错截图，模型要读出错误信息、理解界面状态、结合上下文给出排查步骤。

## 2. 基本结构

图文 VLM 通常由三部分组成。

| 模块 | 作用 |
| --- | --- |
| Vision Encoder | 把图片切块并编码成视觉特征 |
| Projector / Adapter | 把视觉特征映射到 LLM 能接收的表示空间 |
| LLM | 结合文本和视觉表示生成回答 |

简化流程：

```text
image pixels
  -> image patches
  -> vision features
  -> projected visual embeddings
  -> language model context
```

其中 LLM 仍然负责大部分语言理解、指令跟随、推理和回答生成。视觉模块负责把图片内容变成可接入上下文的信息。

### 2.1 Vision Encoder

Vision encoder 是图像侧的编码器，常见做法包括 ViT、CLIP vision tower 或专门训练的视觉编码网络。

它通常会先把图片切成 patch：

```text
图片
  -> 16x16 或 14x14 patch
  -> patch embedding
  -> vision transformer
  -> image features
```

需要注意的是，vision encoder 看到的是经过预处理后的图片，不一定是用户原始上传的完整分辨率。缩放、裁剪、padding、动态分辨率切块都会影响模型能否看清文字、细节和空间关系。

### 2.2 Projector / Adapter

vision encoder 输出的特征不能直接喂给 LLM。两者的向量维度、训练目标和语义空间通常不同，因此需要 projector 或 adapter。

常见形式：

- 线性层：简单把视觉特征映射到 LLM hidden size。
- MLP projector：用多层感知机做非线性映射。
- Q-Former / resampler：用少量查询向量压缩视觉特征。
- Cross-attention adapter：让 LLM 在特定层读取视觉特征。

这个模块决定了视觉信息如何进入语言模型。projector 太弱，模型可能只能识别大概内容；视觉 token 太多，又会增加上下文长度和计算成本。

### 2.3 LLM

LLM 接收文本 token 和视觉 token 后，按自回归方式生成回答。对用户来说，交互像普通聊天：

```text
user:
  <image>
  这张图里有哪些异常？

assistant:
  ...
```

但在模型内部，`<image>` 这类占位符会被展开成一组视觉表示。语言模型生成每个 token 时，可以通过注意力读取这些视觉表示。

## 3. 图像 Token 与视觉占位符

多模态输入通常会在 chat template 中放入特殊占位符，例如：

```text
<image>
请解释这张图。
```

或者使用 OpenAI 风格的结构化消息：

```json
[
  {"type": "image_url", "image_url": {"url": "..."}},
  {"type": "text", "text": "请解释这张图。"}
]
```

推理框架会把图片交给 processor 处理，再把占位符替换或对齐到视觉 token。

这也是多模态部署容易出错的地方：

- chat template 没有正确插入图片占位符。
- tokenizer 不认识模型需要的特殊 token。
- processor 和模型权重不匹配。
- 多图顺序和文本描述顺序不一致。
- 推理框架只加载了语言模型，没有加载视觉模块。

多模态模型不能只看 `.safetensors` 权重是否存在。tokenizer、processor、chat template、special tokens 和模型结构配置必须一起匹配。

## 4. 图片如何变成上下文成本

图片不会免费进入上下文。它通常会被转换成一组视觉 token 或等价的视觉 embedding。

影响成本的因素包括：

| 因素 | 影响 |
| --- | --- |
| 输入分辨率 | 分辨率越高，视觉特征越多，预处理和编码成本越高 |
| 切图策略 | 动态切块能保留细节，但会增加 token 数 |
| 多图数量 | 多张图会线性增加视觉上下文 |
| 视频帧数 | 抽帧越多，成本越高 |
| OCR 细节 | 小字、密集表格需要更高分辨率或额外 OCR |

工程上经常需要在“看得清”和“跑得快”之间权衡：

```text
低分辨率 -> 成本低，但小字和细节容易丢失
高分辨率 -> 信息多，但延迟、显存和费用更高
```

这也是为什么有些模型在普通图片问答上表现不错，但处理发票、表格截图、代码截图时会明显退化。

## 5. 常见能力

### 5.1 OCR 与文档理解

OCR 是多模态模型最常见的能力之一。它不只是识别文字，还常常需要理解版面：

- 标题、正文、脚注。
- 表格行列关系。
- 表单字段和值。
- 印章、签名、票据编号。
- 截图中的菜单、按钮、错误提示。

传统 OCR 往往先识别文字，再把文字交给下游系统。VLM 可以把“文字内容 + 页面布局 + 用户问题”放在一起理解，因此更适合截图问答、票据解释和图文混排文档理解。

但如果任务要求高精度抽取，例如财务票据、合同字段、医疗报告，仍然应该用专门 OCR、版面分析和规则校验做兜底，不要只依赖开放式回答。

### 5.2 图表与表格理解

图表理解要求模型同时识别视觉元素和数值含义：

- 坐标轴、图例、单位。
- 柱状图、折线图、饼图。
- 表格单元格和行列标题。
- 趋势、异常点、最大值和最小值。

常见失败是模型能说出大致趋势，但数字不精确；或者把图例颜色、坐标轴单位看错。因此用于 BI、财务或科研分析时，最好让模型引用可验证数据源，或者把原始表格数据直接提供给模型。

### 5.3 多图输入

多图任务常见于对比、排序和排查：

```text
图 1 是旧版界面，图 2 是新版界面。请列出变化。
```

多图输入要明确图片顺序。提示词里最好直接引用“第一张图”“第二张图”，不要只说“这张图”和“那张图”。如果框架支持图片标签，也可以使用：

```text
<image_1>
<image_2>
请比较 image_1 和 image_2。
```

模型失败时，常见表现是把两张图的内容混在一起，或者只关注其中一张图。

### 5.4 视频输入

视频模型通常不会逐帧读取完整视频，而是先抽帧或提取片段特征：

```text
video
  -> frame sampling
  -> frame encoder
  -> temporal aggregation
  -> LLM
```

关键问题是时间信息：

- 事件发生的先后顺序。
- 动作是否连续。
- 画面变化发生在哪个时间点。
- 音频和画面是否对齐。

如果只抽少量帧，模型可能漏掉短暂事件；如果抽帧太密，成本会迅速上升。视频理解更适合“摘要、粗粒度事件识别、片段定位”，对精确时序和细粒度动作仍需专门模型或规则系统配合。

### 5.5 音频与语音

音频多模态有两种常见路线：

- 级联式：ASR 把语音转文字，再交给 LLM。
- 端到端：音频 encoder 直接接入语言模型。

级联式工程成熟，便于做纠错、字幕、检索和审计。端到端模型更容易保留语气、停顿、情绪、说话人特征等信息，也更适合实时语音助手。

语音任务要额外关注：

- 识别准确率。
- 多语种和口音。
- 噪声环境。
- 说话人分离。
- 流式延迟。
- TTS 输出稳定性。

## 6. 训练流程

多模态模型通常不是从零开始训练所有能力，而是在已有视觉模型和语言模型基础上对齐。

常见阶段：

```text
视觉-语言对齐预训练
  -> 多模态指令微调
  -> 偏好对齐 / 强化学习
  -> 任务专项微调
```

### 6.1 视觉-语言对齐

这一阶段让模型学会把视觉特征和语言空间对齐。训练数据常见于图文对、图片标题、网页图文、OCR 文档等。

目标是让模型知道：

- 图里有哪些对象。
- 图像区域和文字描述如何对应。
- 图片整体语义是什么。
- 视觉信息如何转成可回答的语言信息。

### 6.2 多模态指令微调

指令微调让模型学会按用户问题完成任务，而不只是描述图片。

数据形式类似：

```text
image + user question -> assistant answer
```

任务包括 OCR、VQA、图表问答、复杂推理、拒答、安全判断、多图对比等。高质量指令数据会明显影响模型是否“会用眼睛解决任务”，而不是只生成泛泛描述。

### 6.3 偏好对齐

多模态模型也需要偏好对齐，尤其是减少幻觉和过度猜测。

例如同一张图，两个回答：

```text
A: 图中可能有一只黑色的狗，但图片较模糊，无法确认品种。
B: 图中是一只拉布拉多犬，年龄约 3 岁。
```

如果图片信息不足，A 往往更可靠。偏好对齐要让模型学会在视觉证据不足时表达不确定，而不是编造细节。

## 7. 能力边界与常见失败

多模态模型很容易给人“看懂了”的错觉，但它仍然是生成模型，存在明显边界。

| 失败类型 | 表现 |
| --- | --- |
| 视觉幻觉 | 图中没有的物体、文字或关系被说出来 |
| 小字识别错误 | 截图、票据、表格里的小字读错 |
| 空间关系错误 | 左右、前后、包含关系判断错 |
| 数字不精确 | 图表数值、表格金额、日期识别错 |
| 过度推断 | 从图片猜测身份、年龄、地点、意图 |
| 多图混淆 | 把不同图片中的内容合并 |
| 视频漏检 | 抽帧没覆盖关键瞬间 |

视觉幻觉尤其重要。模型可能会根据常识补全看不清的内容，而不是承认“不确定”。生产系统应该明确要求模型区分：

- 图片中能直接看到的信息。
- 根据上下文推测的信息。
- 无法确认的信息。

## 8. 评估方法

多模态评估要同时评估视觉理解、语言回答和跨模态推理。

常见维度：

| 维度 | 评估重点 |
| --- | --- |
| OCR | 字符准确率、字段准确率、版面理解 |
| VQA | 回答是否基于图片，是否答到问题 |
| 图表理解 | 数值、趋势、单位、图例是否正确 |
| 空间关系 | 位置、方向、数量、包含关系 |
| 多图对比 | 差异是否完整，图片是否混淆 |
| 视频理解 | 时序、事件、动作、片段定位 |
| 安全 | 人脸、隐私、敏感场景和越权推断 |
| 幻觉 | 是否编造不可见内容 |

评估时要记录输入预处理配置：

- 图片原始尺寸。
- resize 和 crop 策略。
- 是否开启动态分辨率。
- 多图顺序。
- 视频抽帧间隔。
- prompt 和 chat template。
- processor、tokenizer、模型 revision。

同一个模型在不同 processor 或推理框架下，结果可能不同。多模态评估如果不记录预处理，很难复现。

## 9. 部署注意事项

部署多模态模型时，除了普通 LLM 的 tokenizer、权重、chat template，还要关注视觉和音频侧组件。

### 9.1 文件要完整

常见必需文件包括：

- 模型结构配置。
- 语言模型权重。
- vision encoder 权重。
- projector / adapter 权重。
- tokenizer 配置。
- special tokens 配置。
- processor / image processor 配置。
- chat template。

只迁移语言模型权重，通常无法保留多模态能力。

### 9.2 推理框架要支持该模型

不是所有推理框架都完整支持每个 VLM。需要确认：

- 是否支持该模型架构。
- 是否支持图片、视频或音频输入。
- 是否支持动态分辨率和多图。
- 是否支持流式输出。
- 是否支持量化后的视觉模块。
- 是否支持 batching 和并发。

有些框架可以部署语言部分，但不能正确处理视觉输入。此时模型看似能启动，实际只是在根据文本提示回答。

### 9.3 Processor、Tokenizer 和 Chat Template 要一致

多模态输入链路可以理解成：

```text
原始图片 / 音频
  -> processor
  -> 视觉或音频特征

文本消息
  -> chat template
  -> tokenizer
  -> token ids

特征 + token ids
  -> model
```

任何一环不一致，都可能导致效果异常。例如：

- 图片占位符数量和视觉特征数量对不上。
- special token id 与模型训练时不同。
- processor 使用了错误的均值、方差、分辨率。
- chat template 把图片放在了模型不习惯的位置。

### 9.4 量化要谨慎

多模态模型量化不只影响语言能力，也可能影响视觉对齐。常见策略是：

- 语言模型部分量化。
- vision encoder 保持 FP16 / BF16。
- projector 保持较高精度。
- 对 OCR、图表和多图任务单独回归测试。

如果只跑普通聊天样本，很难发现视觉能力退化。

## 10. 业务选型建议

选多模态模型时，不要只看通用榜单。建议按场景拆分。

| 场景 | 重点关注 |
| --- | --- |
| 截图问答 | OCR、小字、UI 元素、中文能力 |
| 文档抽取 | 表格、字段准确率、可校验结构化输出 |
| 图表分析 | 数字精度、单位、图例、趋势解释 |
| 多图对比 | 图片顺序、差异召回率、稳定性 |
| 视频摘要 | 抽帧策略、时序理解、长视频成本 |
| 语音助手 | ASR、TTS、流式延迟、打断能力 |
| 机器人 | 视觉定位、动作规划、工具调用、安全边界 |

如果业务要求高精度，推荐组合式架构：

```text
专用 OCR / 检测 / ASR
  -> 结构化结果
  -> LLM / VLM 综合推理
  -> 规则校验 / 人工复核
```

VLM 适合做理解、解释、推理和交互，不应该独自承担所有高风险识别和校验。

## 11. 和 RAG、Agent 的关系

多模态模型经常和 RAG、Agent 结合。

### 11.1 多模态 RAG

多模态 RAG 可以检索图片、页面截图、PDF 页面、视频片段或图文混合 chunk。

常见链路：

```text
用户问题
  -> 检索相关文本 / 图片 / 页面
  -> VLM 阅读证据
  -> 生成带引用的回答
```

关键问题是证据可追踪。模型回答图表或截图内容时，最好能指向原始页面、图片区域或文档页码。

### 11.2 多模态 Agent

多模态 Agent 可以观察屏幕、理解页面状态，再调用工具完成任务。例如：

- 看浏览器截图并点击按钮。
- 根据报错截图修改配置。
- 读取仪表盘截图并生成告警。
- 根据机器人视觉输入选择下一步动作。

这类系统的风险更高，因为视觉误判可能导致错误操作。需要加入权限控制、动作确认、回滚机制和操作日志。

## 12. 实践建议

落地多模态模型时，可以按这个顺序推进：

1. 明确输入类型：图片、截图、PDF 页面、视频还是语音。
2. 建立小型评估集：覆盖真实图片、低清图片、小字、表格和异常样本。
3. 固定预处理：记录 resize、crop、多图顺序和抽帧策略。
4. 先测基础能力：OCR、物体识别、图表、空间关系。
5. 再测业务任务：按真实 prompt 和输出格式评估。
6. 加入失败分类：小字、幻觉、数字、顺序、拒答、格式。
7. 做工程兜底：专用 OCR、规则校验、引用、人审和重试。
8. 上线后持续采样：把真实失败图片补回回归集。

多模态能力看起来直观，但评估和工程化比纯文本更容易被忽略。真正上线时，输入预处理和失败兜底往往和模型本身一样重要。

## 13. 总结

多模态模型的核心是把图像、视频、音频等非文本信息接入语言模型上下文。它通常由 vision encoder、projector / adapter 和 LLM 组成，通过视觉 token 或视觉 embedding 参与生成。

可以抓住四点：

- VLM 不是直接看图，而是通过视觉模块把图片编码后交给 LLM。
- processor、tokenizer、chat template 和特殊 token 是多模态链路的一部分。
- OCR、图表、多图、视频和语音都有各自的预处理和评估问题。
- 多模态模型仍会幻觉，高风险任务必须加入校验、引用和人工复核。

如果只记住一句话：多模态模型把“看见、听见的内容”转成 LLM 能处理的上下文，但它仍然需要严格评估和工程兜底，才能可靠进入生产系统。
