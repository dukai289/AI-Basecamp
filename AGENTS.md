# Repository Guidelines

## Project Structure & Module Organization

This repository is a Docusaurus 3 site for AI-Basecamp. Public documentation lives in `docs/`, with category metadata in `_category_.json` files and autogenerated sidebars configured by `sidebars.js`. Blog/news posts live in `blog/` and should follow the existing date-based pattern, for example `blog/2026-04-15-ai-news.md`. Standalone pages belong in `src/pages/`, shared styling in `src/css/custom.css`, and Docusaurus theme overrides in `src/theme/`. Static assets are served from `static/`, especially `static/img/`. Operational configs and scripts are under `ops/`. Keep unpublished or sensitive drafts in `docs-private/`; they are not part of the public site.

## Build, Test, and Development Commands

Install dependencies with:

```bash
npm install
```

Run the local development server:

```bash
npm run start -- --host 0.0.0.0 --port 3001
```

Build the production site with `npm run build`; Docusaurus treats broken links as build errors. Preview the built output with `npm run serve`. Use `npm run clear` when cached Docusaurus state causes stale pages or sidebar behavior.

## Coding Style & Naming Conventions

Use Markdown or MDX for content pages. Prefer concise Chinese titles and filenames that match the visible topic, preserving existing mixed Chinese/English naming such as `docs/tools-and-frameworks/deployment/vLLM.md`. JavaScript files use CommonJS exports, two-space indentation, semicolons, and `// @ts-check` where already present. Put reusable images in `static/img/` and reference them with Docusaurus paths such as `/img/logo.png`.

## Testing Guidelines

There is no separate test runner configured. Treat `npm run build` as the required validation before submitting changes because it checks Docusaurus compilation, routing, and broken links. For content-only changes, also review the affected page with `npm run start`. When changing citation handling, sidebars, navigation, or theme overrides, verify at least one docs page, one blog page, and `/changelog`.

## Commit & Pull Request Guidelines

Recent commits are short, descriptive, and often Chinese, for example `0415周报` or `修改cron.sh，增加了日志`. Keep commit subjects specific to the changed area, such as `docs: 增加vLLM部署指南` or `ops: 更新nginx 404配置`. Pull requests should include a brief summary, affected paths, validation performed (`npm run build`), and screenshots for visible UI or styling changes. Link related issues or task notes when available.

## Security & Configuration Tips

Do not publish private notes from `docs-private/` without review and sanitization. Avoid committing secrets, internal tokens, raw logs, or unpublished credentials in `ops/`, `info/`, or Markdown examples. Generated output such as `build/` and `.docusaurus/` should normally stay out of manual edits.
